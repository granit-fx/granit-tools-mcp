# ADR-001: JSON Index + Cloudflare Workers

> **Date:** 2026-03-18
> **Authors:** Jean-Francois Meyers
> **Status:** Accepted

## Context

AI assistants (Claude Code, Cursor, Windsurf) need structured access to Granit
documentation to give accurate, contextual answers. The
[Model Context Protocol](https://modelcontextprotocol.io/) (MCP) provides a
standard interface for this: the assistant calls tools exposed by a remote server.

The docs site already generates `llms.txt` / `llms-full.txt` via the
`starlight-llms-txt` plugin. The question is how to build a **searchable** MCP
endpoint on top of this content.

Requirements:

- **Zero ongoing cost** (free-tier hosting)
- **Low latency** (< 500 ms for search)
- **Automatic updates** on each docs deploy
- **Three tools**: full-text search, module reference lookup, pattern listing
- **No external API keys** (no OpenAI/Voyage embedding calls)

## Decision

- **Search backend**: Pre-built JSON index generated at docs build time
- **Runtime**: Cloudflare Workers (Hono + MCP SDK)
- **Transport**: Streamable HTTP (MCP 2025-03-26 spec, stateless)
- **Cache**: Cloudflare KV (24 h TTL on the index)
- **Domain**: `mcp.granit-fx.dev` (Workers custom domain)

The index is generated by `docs-site/scripts/generate-search-index.mjs` in the
`granit-dotnet` repo. It reads all `.md`/`.mdx` source files, extracts
frontmatter + cleaned content, and writes `search-index.json` to `dist/`. It is
deployed alongside the docs on CF Pages.

## Alternatives considered

### Option 1: Pre-built JSON index + Cloudflare Workers (selected)

- Generate structured `search-index.json` at build time (~261 entries, ~1.1 MB)
- Worker fetches index from CF Pages, caches in KV, performs TF-IDF search
- Module lookup by URL slug / title matching
- Pattern listing by category filter

### Option 2: Pagefind (WASM full-text on built HTML)

- **Advantage**: excellent full-text search quality, built for static sites
- **Disadvantage**: WASM binary (~160 KB) requires loading in Worker, complex
  integration, Pagefind API designed for browser not server-side, fragmented
  index files incompatible with KV caching strategy

### Option 3: Vector embeddings (Voyage/OpenAI + Cloudflare Vectorize)

- **Advantage**: semantic search, handles synonyms and rephrased queries
- **Disadvantage**: API cost per embedding call, chunking complexity, cold
  start latency for embedding generation, overkill for ~300 pages of
  well-structured technical documentation

### Option 4: Serve `llms-full.txt` as MCP resources (no search)

- **Advantage**: zero extra infrastructure, already generated
- **Disadvantage**: no search capability, high token consumption (~200 KB raw
  text per request), no structured metadata (categories, URLs)

### Option 5: Deno + Netlify Edge (Astro docs approach)

- **Advantage**: proven approach (Astro docs MCP uses this)
- **Disadvantage**: Astro uses kapa.ai as backend (paid service), Netlify Edge
  has lower free-tier limits than CF Workers (125K req/month vs 100K req/day)

## Justification

| Criterion | JSON + CF | Pagefind | Embeddings | llms-full.txt | Deno + Netlify |
| --------- | --------- | -------- | ---------- | ------------- | -------------- |
| Search quality | Good (TF-IDF) | Excellent | Excellent | None | Depends |
| Cost | Free | Free | ~$5/month | Free | Free (limited) |
| Complexity | Low | High | High | Minimal | Medium |
| Latency | < 100 ms | ~300 ms | ~500 ms | N/A | ~200 ms |
| Auto-update | Yes (CI) | Yes (CI) | Needs re-embed | Yes (CI) | Yes (CI) |
| Sovereignty | EU (CF) | EU (CF) | US API calls | EU (CF) | US (Netlify) |
| Structured metadata | Yes | Partial | Yes | No | Depends |

The JSON index approach gives the best balance: good search quality for
well-structured technical docs, zero cost, minimal complexity, and full data
sovereignty.

## Tools exposed

| Tool | Description |
| ---- | ----------- |
| `search_granit_docs` | TF-IDF search with weighted scoring (title 5x, description 3x, content 1x) |
| `get_module_reference` | Module doc lookup by name, slug, or prefix |
| `list_patterns` | Architecture patterns grouped by platform |

## Consequences

### Positive

- AI assistants get accurate, up-to-date Granit documentation context
- Zero ongoing cost (CF Workers free tier: 100K req/day)
- Index regenerated automatically on every docs deploy
- Stateless Worker — no session management, no database
- Data sovereignty: all processing on Cloudflare EU, no external API calls

### Negative

- TF-IDF search lacks semantic understanding (no synonym matching)
- 1.1 MB index loaded per cold start (mitigated by KV cache)
- Two repos to maintain (`granit-docs-mcp` + index script in `granit-dotnet`)

## Re-evaluation conditions

This decision should be re-evaluated if:

- Documentation corpus exceeds 1000 pages (index size may impact performance)
- User feedback shows TF-IDF quality is insufficient (consider adding embeddings)
- Cloudflare Vectorize free tier becomes available (hybrid search at zero cost)
- MCP specification adds built-in search primitives
